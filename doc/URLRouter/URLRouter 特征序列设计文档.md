# URLRouter 特征序列设计文档

---

## 一、概述

特征序列（Feature Sequence）是 URLRouter 路由匹配阶段的核心数据结构。它将段模式编译为一个元组序列，每个元组描述一个**移动操作**和在该位置需要验证的**关键字**。

匹配阶段只执行特征序列，不关心捕获细节，从而实现高性能的路由匹配。

---

## 二、核心定义

### 2.1 HEAD 与 END

每个段有两个不可变的位置基准：

| 基准 | 定义 | 值 |
|------|------|-----|
| **HEAD** | 段数组的开始位置 | 固定为 0 |
| **END** | 段数组的结束位置 | 运行时确定的段长度 `len` |

**重要说明**：
- `HEAD` 是绝对基准，不随指针移动而改变
- `END` 是运行时值，编译时以符号形式存在
- 所有位置都是相对于 `HEAD` 的绝对坐标

### 2.2 指针

解析指针（cursor）表示当前在段内的位置，初始值为 `HEAD`（即 0）。

指针随操作符执行而移动，但 `HEAD` 和 `END` 始终保持不变。

### 2.3 移动操作

特征序列中的每个元组包含一个**移动操作**，定义从**当前指针位置**到**关键字起始位置**的移动方式。

移动操作的结果位置是绝对坐标，计算公式为：

```
目标位置 = HEAD + 偏移表达式
```

其中偏移表达式可以是：
- 常量：如 `HEAD + 5`
- END 相关：如 `END - 3`

**注意**：虽然 `HEAD = 0` 时常量退化为数值，但语义上仍是基于 `HEAD` 的绝对位置。

## 三、特征元组类型

### 3.1 类型定义

| 元组类型 | 移动语义 | 关键字 | 说明 |
|---------|---------|--------|------|
| `(HEAD + n, kw)` | 移动到 `HEAD + n` | 有 | 绝对位置匹配 |
| `(END - n, kw)` | 移动到 `END - n` | 有 | 从段尾偏移匹配 |
| `('c', kw)` | 向结尾查找 `'c'` | 有 | 查找后匹配 |
| `('<', 'c', kw)` | 向开头查找 `'c'` | 有 | 反向查找后匹配 |
| `(HEAD + n, NULL)` | 移动到 `HEAD + n` | 无 | 纯绝对移动 |
| `(END - n, NULL)` | 移动到 `END - n` | 无 | 纯段尾移动 |
| `('c', NULL)` | 向结尾查找 `'c'` | 无 | 纯查找 |
| `('<', 'c', NULL)` | 向开头查找 `'c'` | 无 | 纯反向查找 |

### 3.2 统一表示

为简化表示，采用以下格式：

| 表示 | 含义 |
|------|------|
| `(n, kw)` | 移动到 `HEAD + n`，匹配关键字 `kw`（`kw` 可为 `NULL`） |
| `(END, kw)` | 移动到 `END`，匹配关键字 `kw` |
| `(END-n, kw)` | 移动到 `END - n`，匹配关键字 `kw` |
| `('c', kw)` | 向结尾查找 `'c'`，匹配关键字 `kw` |
| `('<', 'c', kw)` | 向开头查找 `'c'`，匹配关键字 `kw` |

**注意**：`(n, kw)` 中的 `n` 是相对于 `HEAD` 的绝对坐标，但由于 `HEAD = 0`，数值上等于从段首偏移的距离。

## 四、特征序列的生成

### 4.1 基本规则

特征序列由段模式编译生成，编译过程遵循以下原则：

1. **关键字是唯一锚点**：只有携带关键字的元组参与路由分支区分
2. **连续操作合并**：连续的常量移动合并为一个绝对位置
3. **查询操作独立**：查询操作（VAR）不可合并，作为独立元组输出
4. **END 约束**：`END` 只能与负数相加，纯 `END` 只能出现在序列末尾

### 4.2 移动操作的绝对坐标

所有移动操作的结果位置都是绝对坐标：

- `$[n]` → 移动到 `HEAD + n`
- `$[>n]` → 当前指针 + n
- `$[<n]` → 当前指针 - n
- `$[END]` → 移动到 `END`
- `$[END-n]` → 移动到 `END - n`

### 4.3 示例：绝对坐标的体现

```
模式：$[5]$'key'

指针初始位置：HEAD (0)
$[5]：移动到 HEAD + 5 = 5
$'key'：从位置 5 开始匹配 "key"

特征序列：(5, "key")
```

```
模式：${2}$[>1]$'key'

指针初始位置：HEAD (0)
${2}：移动到 HEAD + 2 = 2
$[>1]：移动到 2 + 1 = 3
$'key'：从位置 3 开始匹配 "key"

特征序列：(3, "key")
```

```
模式：$'prefix'${3}

指针初始位置：HEAD (0)
$'prefix'：匹配 "prefix"，指针移动到 6
${3}：从位置 6 捕获 3 个字符，指针移动到 9

特征序列：(0, "prefix") → (9, NULL)
```

## 五、END 的特殊处理

### 5.1 END 作为符号

`END` 在编译时作为符号保留，运行时替换为段长度 `len`。

### 5.2 END 的相加规则

| 表达式 | 结果 | 约束 |
|--------|------|------|
| `END + n` | 无效 | 正数不允许（不能越过 END） |
| `END - n` | `END - n` | 允许，`n ≥ 0` |
| `END + END` | 无效 | 不允许 |

### 5.3 END 的位置约束

纯 `END`（不带偏移）只能出现在特征序列的末尾：

```
✅ (END, NULL)    — 序列末尾
❌ (END, "kw")    — 操作的语义是检查关键字的起始位位置,关键字的起始位置不可能是结尾
✅ (END-4, "kw")  — 可在任意位置
❌ (END, NULL) 后还有其他元组
```

原因：移动到 `END` 后，后续操作无法保证指针仍在有效范围内。

## 六、关键字与唯一性

### 6.1 关键字的作用

关键字是路由分支区分的**唯一依据**：

- 携带关键字的元组参与路由树的构建和匹配
- 关键字为 `NULL` 的元组（纯移动）不参与分支区分

### 6.2 唯一性保证

两条路由的特征序列在**所有携带关键字的元组**上必须不同，否则视为冲突。

```
规则A：[(3, "key")]
规则B：[(3, "key")]  → 冲突 ❌

规则A：[(0, "v"), ('.', ".")]
规则B：[(0, "v"), ('-', "-")]  → 可区分 ✅
```

### 6.3 示例：查询操作的关键字

```
模式：${'a'}${'b'}$'c'

特征序列：[('a', NULL), ('b', NULL), (0, "c")]

参与分支区分的元组：('a', NULL) 和 ('b', NULL) 的关键字为 NULL，不参与区分
实际只有 (0, "c") 参与区分
```

**注意**：虽然查询操作的关键字为 `NULL`，但查询的**目标字符**隐含在操作类型中，不同的查询字符（如 `'a'` vs `'c'`）本质上是不同的操作，会影响匹配路径。因此特征序列需要保留查询字符信息。

## 七、特征序列示例

### 7.1 基础匹配

```
模式：/$'user'/${}
段1：$'user' → (0, "user")
段2：${} → (END, NULL)

特征序列：
  段0：[(0, "user")]
  段1：[(END, NULL)]
```

### 7.2 偏移合并

```
模式：/${1}${1}${1}$'key'

累计：HEAD + 1 + 1 + 1 = HEAD + 3

特征序列：[(3, "key")]
```

### 7.3 查询与匹配

```
模式：/$'v'${'.'}$'.'${}

操作序列：
  $'v'      → (0, "v")
  ${'.'}    → 查询 '.' → ('.', NULL)
  $'.'      → (0, ".")
  ${}       → (END, NULL)

特征序列：[(0, "v"), ('.', NULL), (0, "."), (END, NULL)]
```

### 7.4 段尾回退

```
模式：/${}$[<4]$'dddd'

操作序列：
  ${}       → 移动到 END
  $[<4]     → END - 4
  $'dddd'   → 匹配 "dddd"

特征序列：[(END-4, "dddd")]
```

### 7.5 复杂混合

```
模式：/${2}${'a'}${'b'}$'c'$[END]$[<4]$'dddd'

特征序列：
  (2, NULL)      ← ${2} 后遇查询
  ('a', NULL)    ← ${'a'}
  ('b', "c")     ← ${'b'} + $'c' 合并
  (END-4, "dddd") ← $[END]$[<4]$'dddd' 合并
```

## 八、数据结构定义

```c
typedef enum {
    FT_ABS_MATCH,     // 绝对位置匹配: (value, keyword)  value = HEAD + n
    FT_END_MATCH,     // END 偏移匹配: (value, keyword)  value = -n 表示 END-n
    FT_FIND_MATCH,    // 查找匹配: (ch, keyword)  向结尾查找
    FT_FIND_REV_MATCH // 反向查找匹配: (ch, keyword) 向开头查找
} feature_type_t;

typedef struct {
    feature_type_t type;
    int value;              // 偏移量或字符 ASCII
    const char *keyword;    // 关键字，可为 NULL
    size_t keyword_len;
} feature_tuple_t;
```

**value 字段语义**：

| type | value 含义 |
|------|-----------|
| `FT_ABS_MATCH` | 绝对位置（`HEAD + value`） |
| `FT_END_MATCH` | 从 END 的偏移（`END - value`），`value ≥ 0` |
| `FT_FIND_MATCH` | 要查找的字符 |
| `FT_FIND_REV_MATCH` | 要查找的字符 |

## 九、设计原则总结

1. **HEAD 与 END 为基准**：所有绝对位置都是相对于 `HEAD` 的绝对坐标
2. **关键字驱动**：只有关键字参与路由分支区分
3. **END 约束**：只能与负数相加，纯 END 只能在末尾
4. **查询独立**：查询操作不可合并，作为独立锚点
5. **偏移合并**：连续的常量移动合并为单一绝对位置
6. **确定性**：特征序列唯一确定匹配路径

## 十、与语法规范的对应

| 语法操作 | 特征元组 |
|---------|---------|
| `$'text'` | `(HEAD + pos, "text")` |
| `${n}`（后跟关键字） | 合并入后续关键字 |
| `${n}`（末尾） | `(HEAD + pos, NULL)` |
| `${'c'}` | `('c', NULL)` |
| `${}` | `(END, NULL)` |
| `$[n]` | 合并入后续操作 |
| `$[>n]` | 合并入后续操作 |
| `$[<n]` | 合并入后续操作 |
| `$[>'c']` | `('c', NULL)` |
| `$[<'c']` | `('<', 'c', NULL)` |
| `$[END]` | `(END, NULL)` |
| `$[END-n]` | `(END-n, NULL)` 或合并入后续关键字 |
