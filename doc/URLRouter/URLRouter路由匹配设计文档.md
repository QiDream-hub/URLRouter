# URLRouter 路由匹配设计文档

## 一、核心设计思想

**匹配与提取分离**：
- 匹配阶段只关心如何移动指针并验证关键字，不关心捕获操作
- 提取阶段在匹配成功后，根据原始模式执行参数捕获

**段内特征序列**：每个段节点用一个特征序列描述，序列元素为（移动操作，关键字）

---

## 二、特征序列元素类型

特征序列中的移动操作类型有三种：

| 类型 | 表示 | 运行时行为 |
|------|------|------------|
| 正向偏移 | 正整数 n | 指针向结尾方向移动 n 个字符 |
| 负向偏移 | 负整数 -n | 指针向开头方向移动 n 个字符 |
| 字符查找 | 单引号字符 'c' | 从当前指针位置开始向结尾方向查找第一个字符 c，指针移动到该字符位置，找不到则失败 |

特殊表示：
- `END` 表示段尾位置，可用于偏移计算（如 `END-2` 表示从段尾向开头方向 2 个字符的位置）
- 关键字可以为空，表示该元组仅移动指针而不验证内容

---

## 三、编译规则

### 3.1 模式到特征序列的转换

编译时顺序处理段模式中的每个操作符，生成特征序列：

**转换规则：**

1. 遍历操作符序列，维护当前累计偏移量（初始为 0）
2. 遇到纯偏移操作（捕获操作、移动操作）时：
   - 正向移动：累计偏移量增加
   - 负向移动：累计偏移量减少
   - 字符查找：将当前累计偏移量（如有）输出为一个特征元组，然后输出字符查找元组，累计偏移量归零
3. 遇到关键字操作（精确匹配）时：
   - 将当前累计偏移量（如有）与关键字组合输出为一个特征元组
   - 累计偏移量归零
   - 指针增加关键字长度（为下一个元组确定基准位置）
4. 模式处理完毕后，如有剩余累计偏移量，输出为（偏移量，NULL）元组

**偏移合并原则：**
- 连续的正向偏移可合并为一个总偏移量
- 正向偏移与负向偏移可相互抵消
- 字符查找操作不可合并，必须单独输出

**示例：**

```
模式: ${1}${1}${1}$'key'
操作: 偏移+1, 偏移+1, 偏移+1, 关键字"key"
累计: 偏移累加为3，遇到关键字输出 (3, "key")
特征序列: [(3, "key")]
```

```
模式: ${b}$'b'${1}$'a'
操作: 字符查找'b', 关键字"b", 偏移+1, 关键字"a"
累计: 输出('b', "b")，偏移累加1后遇到关键字输出 (1, "a")
特征序列: [('b', "b"), (1, "a")]
```

```
模式: ${}$[<2]$'aa'
操作: 偏移到结尾(END), 偏移-2, 关键字"aa"
累计: 输出(END-2, "aa")
特征序列: [(END-2, "aa")]
```

```
模式: ${}
操作: 偏移到结尾(END)
特征序列: [(END, NULL)]
```

```
模式: ${2}$'a'${3}$'b'
操作: 偏移+2, 关键字"a", 偏移+3, 关键字"b"
特征序列: [(2, "a"), (3, "b")]
```

### 3.2 节点合并规则

两个模式在同一段上的特征序列完全相同时，合并到同一个路由树节点。

**示例：**
- `/${1}${1}${1}$'key'` 特征序列为 `[(3, "key")]`
- `/${2}${1}$'key'` 特征序列为 `[(3, "key")]`
- `/${3}$'key'` 特征序列为 `[(3, "key")]`

三个模式生成相同的特征序列，在路由树中共享同一个匹配节点。

### 3.3 冲突检测

冲突条件：两个模式所有段的特征序列完全相同，且段数相同。

处理：编译时报错，拒绝注册。因为相同的 URL 路径无法区分应该执行哪个模式。

段数不同的模式可以共存，因为 URL 长度不同可以自然区分。

### 3.4 段尾对齐

特征序列执行完毕后，指针必须位于段尾。此规则确保模式完整消耗整个段。

**示例：**
- `/${2}`：特征序列 `[(2, NULL)]`，要求段长度必须为 2
- `/${2}$'a'`：特征序列 `[(2, "a")]`，要求段长度为 3（偏移 2 + 关键字长度 1）
- `/${}$'a'`：特征序列 `[(END-1, "a")]`，要求段最后两个字符为"？a"（END-1 位置开始匹配"a"）

### 3.5 语法限制

- 捕获剩余操作（`${}`）后面不能跟任何其他操作，因为指针已到段尾
- 连续偏移操作自动合并
- 关键字必须完全匹配，不支持前缀或后缀模糊匹配
- URL 中不允许出现空段（连续斜杠），编译时报错

## 四、匹配算法

### 4.1 段匹配流程

对每个段执行以下步骤：

1. 初始化指针位置为 0
2. 按顺序遍历特征序列中的每个元组 (移动操作, 关键字)
3. 根据移动操作类型调整指针：
   - 正整数 n：指针向结尾方向移动 n 个字符，移动后位置不能超过段长度
   - 负整数 -n：指针向开头方向移动 n 个字符，移动后位置不能小于 0
   - 字符 'c'：从当前位置开始向结尾方向查找第一个字符 c，指针移动到该字符位置，找不到则匹配失败
   - END 相关：END 表示段长度，END-n 表示段长度减 n
4. 如果当前元组包含关键字：
   - 检查从指针位置开始是否有足够长度容纳关键字
   - 精确比较关键字是否匹配
   - 匹配成功后指针增加关键字长度
5. 所有元组处理完毕后，检查指针是否正好等于段长度，否则匹配失败
6. 所有检查通过则段匹配成功

### 4.2 匹配特性

- 确定性执行，无需回溯
- 时间复杂度与段长度和元组数量成正比
- 同一偏移位置有多个关键字时，按注册顺序依次匹配，不回退
- 指针移动和关键字验证基于当前指针状态，每个元组的起始位置依赖前一元组的执行结果

### 4.3 完整 URL 匹配流程

1. 按 HTTP 方法选择对应的路由树
2. 将 URL 按斜杠分割为段数组
3. 从路由树根节点开始，遍历每个段：
   - 根据段内容在当前节点的子节点中查找匹配项
   - 匹配依据：段内容执行子节点的特征序列后全部成功
   - 找不到匹配则整个路由匹配失败
4. 所有段匹配完成后，到达叶子节点
5. 返回叶子节点（包含提取器和处理函数）

## 五、参数提取

### 5.1 提取器

提取器保存原始模式的完整操作序列，在路由匹配成功后执行。提取器中的操作包括：

| 操作类型 | 行为 |
|----------|------|
| 定长捕获 | 捕获从当前指针开始的指定数量字符，指针向结尾移动指定长度 |
| 捕获到字符 | 捕获从当前指针开始直到遇到指定字符（不包含该字符），指针移动到该字符位置 |
| 捕获剩余 | 捕获从当前指针到段尾的所有字符，指针移动到段尾 |
| 验证字符串 | 验证当前位置是否匹配指定字符串，不产生捕获，指针增加字符串长度 |
| 移动指针 | 将指针向指定方向移动指定距离，不产生捕获 |

### 5.2 提取规则

- 每个段独立执行提取操作，指针从 0 开始
- 提取器按原始操作顺序执行，保留完整的捕获边界信息
- 捕获结果按顺序存储为（指向原始 URL 的指针，长度）元组
- 提取失败表示编译或数据错误，整个路由匹配失败
- 不回退尝试其他模式，因为匹配阶段已确保段内容符合约束

### 5.3 编译时优化

由于匹配阶段已验证所有关键字的存在，提取器中的验证字符串操作可转换为移动指针操作，跳过已验证的内容，提升提取性能。

## 六、HTTP 方法处理

### 6.1 设计决策

路由树不存储 HTTP 方法相关信息。每个 HTTP 方法拥有独立的路由树，按方法索引组织。

### 6.2 方法集合

支持的标准 HTTP 方法包括：
- GET
- POST
- PUT
- DELETE
- PATCH
- HEAD
- OPTIONS

### 6.3 操作流程

注册路由时：
- 根据指定的 HTTP 方法选择对应的路由树
- 将模式编译后插入到该树中

匹配路由时：
- 根据请求的 HTTP 方法选择对应的路由树
- 在该树中执行 URL 匹配
- 返回匹配到的叶子节点

### 6.4 设计优势

- 不同方法的相同路径互不干扰
- 匹配时无需检查方法，性能更高
- 节点结构更简单，不关心 HTTP 方法
- 每种方法可独立管理路由规则

---

## 七、设计规则总结

| 规则 | 说明 |
|------|------|
| 匹配与提取分离 | 匹配阶段不关心捕获，只验证移动和关键字 |
| 特征序列驱动 | 每个段用（移动操作，关键字）序列描述 |
| 移动操作类型 | 支持正向偏移、负向偏移、字符查找 |
| 偏移合并 | 连续偏移操作合并为单个元组 |
| 段尾必须对齐 | 特征序列执行后指针必须正好在段尾 |
| 合并相同特征 | 特征序列完全相同的段合并为一个节点 |
| 冲突检测 | 完整路径（段数+每段特征）相同时编译报错 |
| 重叠关键字 | 按注册顺序匹配，不回退 |
| 空段禁止 | URL 中连续斜杠编译报错 |
| 提取失败即错误 | 不回退尝试其他模式 |
| 方法隔离 | 每个 HTTP 方法独立的路由树 |

## 八、设计示例

### 8.1 基础路由

模式：`/user/{}`

- 段0：精确字符串"user"
  - 操作：关键字"user"
  - 特征序列：`[(0, "user")]`
- 段1：捕获剩余
  - 操作：偏移到结尾
  - 特征序列：`[(END, NULL)]`

URL：`/user/alice`

- 段0：指针0，验证"user"，指针移动到4，等于段长度 → 成功
- 段1：指针移动到段尾（位置5），等于段长度 → 成功
- 提取参数：["alice"]

### 8.2 定长限制

模式：`/{2}`

- 段0：定长捕获2个字符
  - 特征序列：`[(2, NULL)]`

URL：`/ab` → 段长度2，指针移动2后等于段长度 → 成功
URL：`/abc` → 段长度3，指针移动2后不等于段长度 → 失败

### 8.3 带关键字的定长捕获

模式：`/{2}key`

- 段0：定长捕获2个字符后跟字符串"key"
  - 特征序列：`[(2, "key")]`

URL：`/abkey` → 指针移动2，验证"key"，指针移动3到达段尾 → 成功
提取参数：["ab"]

### 8.4 字符查找

模式：`/{=}=`

- 段0：捕获到'='后跟'='
  - 操作：捕获到'='，关键字"="
  - 特征序列：`[('=', "=")]`

URL：`/name=alice` → 查找'='定位到位置4，验证"="，指针移动1到达位置5 → 成功
提取参数：["name"]

### 8.5 从结尾偏移

模式：`/{}[<2]aa`

- 段0：捕获剩余，回退2，匹配"aa"
  - 特征序列：`[(END-2, "aa")]`

URL：`/xyzabcaa` → 段长度8，END-2=6，指针移动6，验证"aa"，指针移动2到达段尾 → 成功
提取参数：["xyzabc"]

### 8.6 合并示例

模式A：`/{2}key/a`
模式B：`/[>2]key/a`

- 两个模式段0特征序列均为 `[(2, "key")]`，段1特征序列均为 `[(0, "a")]`
- 完整路径相同（2个段，每段特征相同），编译时报冲突错误

### 8.7 合法共存

模式C：`/{2}key`
模式D：`/{2}key/a`

- 模式C只有1个段，模式D有2个段
- 段数不同，可以共存
- URL `/abkey` 匹配模式C
- URL `/abkey/a` 匹配模式D

### 8.8 复杂特征序列

模式：`/${b}$'b'${1}$'a'`

- 段0特征序列：`[('b', "b"), (1, "a")]`

URL：`/xbayca` → 查找'b'定位到位置1，验证"b"，指针移动1到位置2，移动1到位置3，验证"a" → 成功
提取参数：["x", "yc"]（捕获操作产生的参数）
