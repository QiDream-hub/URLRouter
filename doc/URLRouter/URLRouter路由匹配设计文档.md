# URLRouter 路由匹配设计文档

## 一、核心设计思想

**匹配与提取分离**：
- 匹配阶段只关心如何移动指针并验证关键字，不关心捕获操作
- 提取阶段在匹配成功后，根据原始模式执行参数捕获

**段内特征序列**：每个段节点用一个特征序列描述，序列元素为（偏移，关键字）

---

## 二、特征序列元素类型

| 类型 | 表示 | 运行时行为 |
|------|------|------------|
| 固定偏移 | 整数 n | 指针向前移动 n 个位置 |
| 字符锚点 | 字符 c | 指针移动到当前之后第一个字符 c 的位置，找不到则失败 |
| 从结尾偏移 | 负数 -n | 指针移动到从段尾倒数第 n 个字符的位置 |
| 到结尾 | END | 指针移动到段尾 |

---

## 三、编译规则

### 3.1 模式到特征序列的转换

| 模式语法 | 特征序列 | 说明 |
|----------|----------|------|
| 精确字符串 | （0，字符串） | 从段首开始精确匹配该字符串 |
| 定长捕获 | （长度，空） | 要求段长度等于该长度 |
| 捕获剩余 | （END，空） | 捕获从当前位置到段尾的所有字符 |
| 定长捕获后跟字符串 | （长度，字符串） | 跳过指定长度后匹配字符串 |
| 捕获到字符后跟字符串 | （字符，字符串） | 移动到第一个指定字符后匹配字符串 |
| 字符串后跟捕获到字符再跟字符串 | （0，字符串1），（字符，字符串2） | 先匹配字符串1，再移动到字符后匹配字符串2 |
| 跳过指定长度后跟字符串 | （长度，字符串） | 与定长捕获后跟字符串特征相同，可合并 |
| 从结尾回退后跟字符串 | （-长度，字符串） | 从段尾回退指定长度后匹配字符串 |
| 绝对跳转后跟字符串 | （长度，字符串） | 与跳过指定长度特征相同，可合并 |
| 连续定长捕获 | （长度之和，空或字符串） | 多个连续定长捕获合并为一个偏移 |

### 3.2 合并规则

两个模式在同一段上的特征序列完全相同时，合并到同一个路由树节点。

示例：
- 定长捕获后跟字符串与跳过指定长度后跟字符串生成相同的特征序列，可以合并
- 连续定长捕获与单个定长捕获生成相同的特征序列，可以合并
- 不同长度的定长捕获生成不同的特征序列，不能合并

### 3.3 冲突检测

冲突条件：两个模式所有段的特征序列完全相同，且段数相同。

处理：编译时报错，拒绝注册。因为相同的URL路径无法区分应该执行哪个模式。

段数不同的模式可以共存，因为URL长度不同可以自然区分。

### 3.4 语法限制

- 捕获剩余操作后面不能跟任何其他操作，因为指针已到段尾
- 连续定长捕获自动合并为一个偏移
- 关键字必须完全匹配，不支持前缀或后缀模糊匹配
- URL中不允许出现空段（连续斜杠），编译时报错

## 四、匹配算法

### 4.1 段匹配流程

对每个段执行以下步骤：

1. 初始化指针位置为0
2. 按顺序遍历特征序列中的每个元素
3. 根据偏移类型移动指针：
   - 固定偏移：指针增加指定数值
   - 字符锚点：查找当前位置之后第一个指定字符，指针移动到该字符位置
   - 从结尾偏移：指针移动到段长度减去指定数值的位置
   - 到结尾：指针移动到段长度
4. 检查指针是否超出段长度，超出则匹配失败
5. 如果当前元素有关键字：
   - 检查从指针位置开始是否有足够长度容纳关键字
   - 精确比较关键字是否匹配
   - 匹配成功后指针增加关键字长度
6. 所有元素处理完毕后，检查指针是否正好在段尾，否则匹配失败
7. 所有检查通过则段匹配成功

### 4.2 匹配特性

- 确定性执行，无需回溯
- 时间复杂度与段长度和步骤数成正比
- 同一偏移位置有多个关键字时，按注册顺序依次匹配，不回退

### 4.3 完整URL匹配流程

1. 按HTTP方法选择对应的路由树
2. 将URL按斜杠分割为段数组
3. 从路由树根节点开始，遍历每个段：
   - 根据段内容和特征序列在当前节点的子节点中查找匹配项
   - 找不到匹配则整个路由匹配失败
4. 所有段匹配完成后，到达叶子节点
5. 执行叶子节点存储的提取器，得到参数列表
6. 返回对应的处理函数和参数

---

## 五、参数提取

### 5.1 提取器

提取器保存原始模式的完整操作序列，在路由匹配成功后执行。提取器中的操作包括：

| 操作类型 | 行为 |
|----------|------|
| 定长捕获 | 捕获从当前指针开始的指定数量字符 |
| 捕获到字符 | 捕获从当前指针开始直到遇到指定字符（不包含该字符） |
| 捕获剩余 | 捕获从当前指针到段尾的所有字符 |
| 验证字符串 | 验证当前位置是否匹配指定字符串，不产生捕获 |
| 移动指针 | 将指针向前或向后移动指定距离，不产生捕获 |

### 5.2 提取规则

- 匹配到叶子节点后，直接执行该节点对应的提取器
- 提取失败表示编译或数据错误，整个路由匹配失败
- 不回退尝试其他模式，因为匹配阶段已确保段内容符合约束

---

## 六、HTTP方法处理

### 6.1 设计决策

路由树不存储HTTP方法相关信息。每个HTTP方法拥有独立的路由树，按方法索引组织。

### 6.2 方法集合

支持的标准HTTP方法包括：
- GET
- POST
- PUT
- DELETE
- PATCH
- HEAD
- OPTIONS

### 6.3 操作流程

注册路由时：
- 根据指定的HTTP方法选择对应的路由树
- 将模式编译后插入到该树中

匹配路由时：
- 根据请求的HTTP方法选择对应的路由树
- 在该树中执行URL匹配
- 返回匹配到的处理函数和参数

### 6.4 设计优势

- 不同方法的相同路径互不干扰
- 匹配时无需检查方法，性能更高
- 节点结构更简单，不关心HTTP方法
- 每种方法可独立管理路由规则

---

## 七、设计规则总结

| 规则 | 说明 |
|------|------|
| 匹配与提取分离 | 匹配阶段不关心捕获，只验证移动和关键字 |
| 特征序列驱动 | 每个段用（偏移，关键字）序列描述 |
| 段尾必须对齐 | 特征序列执行后指针必须正好在段尾 |
| 合并相同特征 | 特征序列完全相同的段合并为一个节点 |
| 冲突检测 | 完整路径相同时编译报错 |
| 重叠关键字 | 按注册顺序匹配，不回退 |
| 空段禁止 | URL中连续斜杠编译报错 |
| 提取失败即错误 | 不回退尝试其他模式 |
| 方法隔离 | 每个HTTP方法独立的路由树 |

## 八、设计示例

### 8.1 基础路由

模式：`/user/{}`

- 段0：精确字符串"user"，特征序列为（0，"user"）
- 段1：捕获剩余，特征序列为（END，空）

URL：`/user/alice`

- 段0匹配"user"成功
- 段1捕获"alice"成功
- 提取参数：["alice"]

### 8.2 带关键字的捕获

模式：`/{2}key`

- 段0：定长捕获2个字符后跟字符串"key"，特征序列为（2，"key"）

URL：`/abkey`

- 段0：跳过2个字符到达位置2，匹配"key"成功，指针移动到段尾
- 提取参数：["ab"]

### 8.3 合并示例

模式A：`/{2}key/a`
模式B：`/[>2]key/a`

- 两个模式段0特征序列均为（2，"key"），段1特征序列均为（0，"a"）
- 完整路径相同，编译时报冲突错误

### 8.4 合法共存

模式C：`/{2}key`
模式D：`/{2}key/a`

- 模式C只有1个段，模式D有2个段
- 段数不同，可以共存
- URL `/abkey` 匹配模式C
- URL `/abkey/a` 匹配模式D
